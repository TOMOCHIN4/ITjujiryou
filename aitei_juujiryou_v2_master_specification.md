# 愛帝十字陵 v2 包括的変更指示書

最終更新: 2026-05-13

本書は IT十字陵 v1 → v2 への移行を実装者に指示するための包括的な仕様書です。設計思想、実装要件、Phase 構成、ベンチマーク計画、責任分担をすべて含みます。本書 1 本で v2 着手に必要な情報が揃うように構成しています。

---

## 第1部: 設計思想

### 1.1 目的

最終制作物の高品質化。**`Opus 4.7 + 厳選SKILLS` 単体構成を上回る納品物を出すこと** が v2 の存在意義。これを下回るなら v2 は不要。

### 1.2 中核の優位性

**コンテキストの分離**。5つの独立 Claude Code プロセスにより、単体運用では避けられない長い案件における文脈混線を構造で抑え込む。

### 1.3 副次的な価値

ペルソナによる役割の固定。各キャラの気質が品質ガードレールとして機能する。ただし**ペルソナは目的ではなく手段**である点に注意。品質ゲインが出なければペルソナ自体が無意味になる。

## 第2部: アーキテクチャ

### 2.1 5キャラ編成

| キャラ | 役割 | 階層 |
|---|---|---|
| サザン | 社長・儀礼承認・裏側ユーティリティプロセッサ | 管理層 |
| ユウコ | 秘書・窓口・指揮・記憶門番 | 管理層 |
| ハオウ | 特化型エージェント | 実行層 |
| トシ | 特化型エージェント | 実行層 |
| センシロウ | 特化型エージェント | 実行層 |

三兄弟の専門領域 (デザイナー/エンジニア/ライター) は既存実装に従う。

### 2.2 コミュニケーション・トポロジー

- ユウコ ↔ 顧客 (窓口)
- ユウコ ↔ サザン (報告と儀礼承認)
- ユウコ ↔ 三兄弟 (指示と応答)
- **三兄弟 ↔ サザン 間の直接通信は存在しない** (絶対原則)

### 2.3 記憶システム

#### 2.3.1 記憶の二系統

- **会社記憶** (`data/memory/company/`): 全員共通の経験蓄積
- **個人記憶** (`data/memory/{role}/`): 本人専用の経験蓄積

各キャラに個人記憶ディレクトリを用意:
```
data/memory/company/
data/memory/yuko/
data/memory/sazan/
data/memory/haou/
data/memory/toshi/
data/memory/senshirou/
```

`~/.claude/projects/.../memory/MEMORY.md` (Claude Code 組み込み auto-memory) は人間の私的領域として継続。エージェント記憶ネットワークとは別レイヤー。

#### 2.3.2 アクセス権マトリクス

| キャラ | 会社記憶 | 自身の個人記憶 | 他人の個人記憶 |
|---|---|---|---|
| ユウコ | 読み (subagent 経由) | 読み書き | **全閲覧可** (subagent 経由) |
| サザン | 書き (HOOK経由) | 読み書き | 不可 |
| 三兄弟 | 読み (subagent 経由) | 読み書き | 不可 |

**重要**: 「読み」アクセスはすべて記憶検索 subagent 経由のみ。直接 Read は禁止 (詳細は第3部 3.4)。

## 第3部: 主要設計判断

### 3.1 ユウコの3セッション分割

ユウコは案件ライフサイクルで3つの独立セッションに分割される:

| セッション | 役割 | 受け取る情報 | 出力 |
|---|---|---|---|
| **受注/初期案** | grill-me で要件確定、イニシャルプラン作成 | 顧客発言 + 全記憶検索結果 + WF庫情報 | initial_plan.md |
| **統合** | 兄弟3名のレビュー応答を統合してファイナルプラン確定、サザン儀礼承認を受ける | initial_plan.md + 3者のレビュー応答 | final_plan.md |
| **振り分け/監督** | マクロフロー + 各兄弟のミクロワークフローを層構造で保持し、兄弟プロセスへ仕事投下、成果物集約 | final_plan.md + 実行状態 | 納品物 |

セッション間はファイル経由でハンドオフ。各セッションは前段成果物のみを起動時に読み込む。**全記憶を再ロードしない**ことでユウコ側のコンテキスト膨張を構造的に抑制する。

### 3.2 三兄弟のレビュー/実行セッション分離

各兄弟は案件中に少なくとも2セッション (レビュー + 実行) を別プロセスで持つ。実行が複数フェーズに分かれる場合はそのたびに別セッション。

| 段階 | 入力 | 出力 | 個性 |
|---|---|---|---|
| レビュー | マクロフロー全体 + 自分のミクロWF候補 + 検索済み記憶 | 承認 or 修正案 + **理由を self/memory に書き込み** | 反映する |
| 実行 | ファイナルプランから自分の担当ノードのみ + 検索済み記憶 (レビュー時の自分の意図を含む) | 成果物 | 出さず計画通りに動く |

レビュー → 実行ハンドオフは **self/memory 経由**。レビュー段階で書いた修正提案理由が、実行段階の自分にとって判断材料になる。

### 3.3 サザンの二重構造

**表向き** (ユーザーから見えるペルソナ):
- 制圧前進あるのみ。ユウコの計画に絶対反対しない
- 三兄弟と直接コミュニケーションを取らない (ユウコとだけ対話)
- 発言は「ユウコからの伝令に対し、サウザー名台詞21選から1つを選んでオマージュ応答」のみ
- 具体的な検討・修正・提案は一切しない

**裏側** (内部実装、見えない):
- ユウコ → サザン報告のタイミングを契機として、案件全体で必ず発生する単純タスクを担当
- 確実な分担: `company_memory_write` 反映 (HOOK 経由)
- その他の分担は実装側が既存ログから逆引き調査して洗い出し (第3部 3.5 参照)

**含意**:
- v0.13 brevity hook (英語化禁止 + cap 注入) は **撤廃** (新ペルソナでは不要)
- サザンセッションは「伝令の種別」に応じて対応する内部 HOOK が発火する設計
- 発言はあくまで儀礼、実処理は HOOK 群

### 3.4 記憶検索 subagent と物理ガード

#### 3.4.1 記憶検索 subagent

すべての記憶アクセスは専用 subagent 経由で行う。MCP ではなく **subagent として実装** する理由:

- **コンテキスト分離の徹底**: subagent 内で検索 + 要約 + 関連度評価が完結し、呼び出し元には蒸留された要約だけが返る
- **判断を含められる**: 単純 grep ではなく「この案件に関連する過去経験は何か」という意味検索なので、検索 → 関連度評価 → 要約 → 矛盾検出までを自律的に行う
- **キャラ別の検索範囲**: ユウコ用 (全記憶) と兄弟用 (会社 + 自分の個人) で検索範囲が異なるため、キャラごとに別 subagent を定義
- **改善ループ**: MD でプロンプト調整できる

#### 3.4.2 使用必須タイミング (CLAUDE.md で強制)

- ユウコの受注/初期案セッション開始時 (顧客発言を入力に関連過去案件を検索)
- ユウコの統合セッション開始時 (3者応答の整合性確認のため過去類似ケースを検索)
- 兄弟のレビューセッション開始時 (与えられたタスクに関連する自分の経験を検索)
- 兄弟の実行セッション開始時 (担当ノードに関連する自分の経験を検索)

#### 3.4.3 物理ガード

CLAUDE.md ルール (お願いベース) だけでは不十分。**`.claude/settings.json` で Read 権限を物理的に制限**する:

- **三兄弟**: `data/memory/{role}/` 配下のみ Read 許可。`data/memory/company/` および他人の個人記憶への直接 Read を deny。Task tool は allow (subagent 経由なら可)。これで「直接読む = 物理的に不可、subagent 経由が唯一の経路」になる。
- **ユウコ**: Read を全許可で残す。CLAUDE.md ルールで「セッション開始時は記憶検索 subagent を必ず通せ」と規律ベースで強制。設計書 1.2 の全閲覧権を維持するため。

### 3.5 サザン HOOK の棚卸し (実装側調査タスク)

ユウコ → サザン報告タイミングの既存ルールに基づき、各タイミングで割り当てられる HOOK を逆引きで列挙する。実装側既往の予備調査:

| タイミング | 物理的に起きること | HOOK 候補 |
|---|---|---|
| 新規案件受注承認上申 | tasks 行追加、client_request 確定 | client 入力サニタイズ + 過去類似案件検索 (将来) |
| 値引き/納期短縮の裁定 | クライアント再交渉メッセージ受信 | 純儀礼 or 値引き許諾履歴記録 (頻度を見て判断) |
| 品質基準で迷う成果物の上申 | revisions に escalate_to_president 行追加 | 直近 N 件の判断履歴を company/memory に集約 |
| ファイナルプラン承認 (v2新規) | プラン MD が確定 | プラン MD の構造バリデーション (frontmatter 必須項目チェック) |
| company 提案承認 (v2新規) | ユウコがクリーン提案をキューに置く | `company_memory_write` 反映 |
| 案件完了報告 (v2新規) | deliver 完了 | 整理 HOOK 起動 (各兄弟に self/memory 整理を促す) |

Phase 0 で過去 1-2 案件分のログを掘り、上記の検証と追加候補を確定する。

## 第4部: 実装要件

### 4.1 セッション起動形: (α) claude -p 都度起動

**ステートレス短命プロセス** を採用。各セッションは `claude -p <prompt>` で都度起動し、完了とともに終了する。

採用理由:
- v2 の核 (コンテキスト分離) を OS レベルで保証
- `/clear` 同 pane 方式は Claude Code バージョン依存で信頼できない
- pane 10 個並列方式は Max OAuth と画面で破綻

pane 可視性は失われるため、UI 側で「ユウコは今〇〇セッション中」を表示する仕組みが必要 (別 issue として後追い)。

### 4.2 セッション間ハンドオフ

既存の inbox/outbox ファイル基盤を流用:

- ユウコの3セッション間: 前セッションが所定パスに成果物 (initial_plan, integrated_plan, final_plan) を書き、次セッションが起動時に読む
- 兄弟のレビュー → 実行: self/memory 経由
- ユウコ → 兄弟: タスクキュー経由
- 兄弟 → ユウコ: reports 経由

### 4.3 ワークフローとファイナルプランの形式

**カスタムスキーマは発明しない**。すべて Claude Code ネイティブの Markdown + YAML frontmatter 形式。

```markdown
---
name: workflow-or-plan-name
description: 短い説明
case-id: 2026-05-13-client-name  # 案件ファイル時のみ
agents: [haou, toshi, senshirou]  # ファイナルプラン時のみ
workflows-referenced: [...]
---

# 本文 (自然言語のマクロフロー or ワークフロー記述)
```

CC Workflow Studio (VSCode 拡張) との互換性のため、frontmatter フィールドは Studio エクスポート形式に揃える。**Studio の採用は v2 で必須ではない** が、将来採用時の移行コストをゼロにするためスキーマは合わせる。

### 4.4 パーサ・実行エンジンは作らない

ユウコの振り分けセッションがファイナルプランの本文を読み、自然言語のマクロ記述を解釈して兄弟プロセスへ仕事を投下する。LLM が解釈し、状態管理は既存のファイルベース基盤に任せる。専用パーサや実行エンジンは作らない。

### 4.5 ワークフロー倉庫の3階層

| 階層 | パス | 用途 | 改変可否 |
|---|---|---|---|
| 原本 | `workflows/originals/` | Studio 出力 (or 人間手書き)、改変禁止 | **改変禁止** |
| 案件アレンジ | `workflows/cases/{案件ID}/` | 案件ごとのアレンジ、案件スナップショット | 案件中のみ書き換え |
| 蒸留 | `workflows/distilled/` | 案件後の昇格版、再利用可 | 昇格時のみ書き込み |

#### 4.5.1 昇格規律

- 案件終了の整理 HOOK で各兄弟が「再利用価値あり」と判定した最大1本を昇格提案
- ユウコが承認したものが `distilled/` へ昇格
- **蒸留昇格は1案件・1兄弟あたり最大1本**
- `distilled/` は同じ原本から派生3本上限。4本目を作る時は1本退役

#### 4.5.2 肥大化制御

- 原本は触らない。Studio バージョンアップは `v3` → `v4` として共存。90日無参照で削除候補に
- 案件アレンジは90日でアーカイブ (物理削除はせず `cases/_archive/` へ移動)

### 4.6 consult_souther の運用ルール

同期 polling は維持。ただし新ルールを明文化:

- **応答受領時点で HOOK 完了を待たない**
- ユウコは応答を受け取ったら次セッションに進める
- 時間軸前提: ユウコの統合 → 振り分け移行までに `company_memory_write` HOOK は完了している想定
- **HOOK 失敗の安全網**: 次セッション開始時に `data/memory/company/_last_write.log` のタイムスタンプをチェックし、想定範囲外なら警告を発する仕組みを入れる

### 4.7 整理 HOOK の発火タイミング

`SessionEnd` フックは兄弟セッションが長命の場合発火しないため使えない。代わりに **deliver 完了を契機に別プロセスで起動** する形式を採用。`scripts/post_deliver_hook.py` のような形で、deliver イベント発火後に各兄弟役を順次起動する。

### 4.8 Workflow Studio 連携

**v2 着手時点では採用しない**。Phase 2 では人間が手書き MD で `workflows/originals/` に配置する。Studio の採用は Phase 6 以降の検討事項。スキーマだけ揃えておくことで、将来採用時の移行コストをゼロにする。

業務時間外に人間が操作するルールは Studio 採用時に適用。AI エージェントには Studio を操作させない。

```

  ↓
ユウコ「受注/初期案」セッション (claude -p)

  ↓
三兄弟「レビュー」セッション × 3 (claude -p, 並列)
  ・ 各自記憶検索 subagent で関連経験検索
  ・ 承認 or 修正案 + 理由を self/memory に書込
  ↓
ユウコ「統合」セッション (claude -p)
  ・ initial + 3者応答を読み込み
  ・ 矛盾解決・粒度調整
  ・ final_plan.md 出力
  ↓
サザン儀礼承認「制圧前進」
  ↓ (応答時点で HOOK 完了は待たない)
ユウコ「振り分け/監督」セッション (claude -p)
  ・ final_plan.md を解釈
  ・ 兄弟タスクキューに投下 (直列/並列/ループの混合)
  ↓
三兄弟「実行」セッション × N (claude -p)
  ・ 各自記憶検索で関連経験 (レビュー時の自分の意図含む) を検索
  ・ ワークフロー実行、計画通りに動く
  ↓
納品 → 顧客
  ↓
各兄弟「整理 HOOK」起動 (post_deliver_hook.py)
  ・ self/memory 追記
  ・ company 提案ファイル生成
  ・ 蒸留昇格提案 (最大1本)
  ↓
ユウコ統合・バリデーション (門番)
  ・ フォーマット検証
  ・ 矛盾解決・差戻し
  ・ クリーン提案を pending に置く
  ↓
サザン儀礼承認 → company_memory_write HOOK → 物理反映
```

### 8.3 SkillCollection の取り扱い (重要)

`/Users/tomohiro/Desktop/ClaudeCode/SkillCollection` フォルダ内の Skill は、**単体構成と IT十字陵 の両方で自由に使用可能**。

- 評価書 c で言及した「厳選SKILLS」の実体がこのフォルダ
- 両構成が同じスキルセットにアクセスできる状態にする
- これにより「スキルの違いによる差」を排除し、**構造の違いによる差**だけを測る公正な比較になる
