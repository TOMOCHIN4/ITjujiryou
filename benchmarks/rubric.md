# 愛帝十字陵 v2 ベンチマーク評価ルーブリック

仕様書 §4.1「最終制作物の高品質化」を測るための共通ルーブリック。各 Run 終了後に同じルーブリックで採点し、構造の違いによる品質差を切り分ける。

---

## 評価軸 (5 軸 × 5 点満点 = 25 点)

### 軸 1: 要件適合度 (0-5)

> client_persona §8.3 の 14 項目要件のうち、どれだけが納品物 (LP/パンフ/ポスター 等) で **実体として満たされている** か

| スコア | 判定基準 |
|---|---|
| 5 | 14 項目すべてが納品物 + 申し送り書 で網羅。優先順位 4 本も明示的に反映 |
| 4 | 11〜13 項目が網羅。残り 1〜3 項目は申し送り書で「未確定理由」明記 |
| 3 | 8〜10 項目が網羅。漏れ項目があっても主要訴求軸 4 本は通っている |
| 2 | 5〜7 項目のみ。訴求軸 4 本のうち 1 本以上が抜けている |
| 1 | 1〜4 項目のみ。テンプレ的な歯科求人になっている |
| 0 | ターゲット (徳島大歯学部口腔保健科) が誤解されている、または納品物がそもそも要件と無関係 |

採点根拠の明示: 14 項目チェックリストで「○/△/×」を記録

---

### 軸 2: 申し送り精度 (0-5)

> 納品時にクライアントへの **申し送り書 (introduction / handoff)** がどれだけ親切・正確・実務的か

| スコア | 判定基準 |
|---|---|
| 5 | 各納品物の意図・前提条件 (色 / トーン / 想定読者) / 修正フロー / 印刷入稿時の注意 / 求人法令の留意点 / 後続施策の提案、全部書かれている |
| 4 | 上記 7 項目中 5〜6 項目をカバー |
| 3 | 上記 7 項目中 3〜4 項目。各納品物の意図と前提条件は最低限ある |
| 2 | 単に「納品しました」程度のメール。意図や前提が説明されていない |
| 1 | メールがそもそも素っ気ない or 礼を失している |
| 0 | 申し送り書が存在しない / 納品物だけ放り投げ |

---

### 軸 3: 文字数・形式制約遵守 (0-5)

> 自ら立てた制約 (LP の文字数 / パンフのページ構成 / ポスターの判型 / コピーの文字数) を **自分で守り切れたか**

| スコア | 判定基準 |
|---|---|
| 5 | 全制約を遵守。文字数制限・判型・ファイル形式 (HTML / Markdown / 画像) すべて正確 |
| 4 | 制約遵守率 80% 以上。逸脱は理由付きで申し送り |
| 3 | 制約遵守率 60〜79%。一部の文字数オーバーや形式違反あり |
| 2 | 制約遵守率 40〜59%。「○○字程度」と書いて 1.5 倍超過などが散見 |
| 1 | 制約遵守率 20〜39% |
| 0 | 制約をそもそも意識していない / 文字数も判型も無視 |

---

### 軸 4: 誤り・矛盾の少なさ (0-5)

> 事実誤認、内部矛盾、求人法令違反、ペルソナ漏れ、転居前提と矛盾するコピー など

減点項目:
- ペルソナ漏れ (聖帝 / サウザー / 下郎 等) … **致命**: -3 点
- 求人法令違反 (年齢制限の明記、性別指定など) … 軽 1 件 = -1 点
- 院長プロフィール誤り (大阪大卒 / 2008 年卒 / 樟葉所在地 等) … 1 件 = -1 点
- 転居前提と矛盾するコピー (「地元で働ける」など徳島学生向けに不適切) … 1 件 = -2 点
- 内部矛盾 (LP と パンフで採用条件がズレる等) … 1 件 = -1 点
- 文法・誤字脱字 … 軽微なら集計、5 件で -1 点

| スコア | 判定基準 |
|---|---|
| 5 | 上記減点項目ゼロ |
| 4 | 軽微な減点が合計 -1 点以内 |
| 3 | 軽微な減点が合計 -2 点以内 |
| 2 | 致命減点なし、軽微で -3 点〜-4 点 |
| 1 | 致命減点 1 件、または軽微で -5 点以上 |
| 0 | 致命減点 2 件以上 (ペルソナ漏れ + 求人法令違反 等) |

---

### 軸 5: 納品物のそのまま使える度 (0-5)

> クライアントが受け取ったものが **追加作業なしでクリエイティブとして実運用に耐えるか**

| スコア | 判定基準 |
|---|---|
| 5 | LP は HTML/CSS で表示可、パンフ・ポスターは入稿前提を満たす画像 + 文章セット、コピー類は校了レベル。デザイナーが受け取って即印刷会社に回せる |
| 4 | 軽微な手直し (色調整 / 画像置換 1〜2 箇所) で実運用可 |
| 3 | コピーとレイアウト指示はあるが、最終データ化はクライアント側作業が必要 (= プロト納品) |
| 2 | テキスト草稿レベル。デザインは方向性のみ。実装は未着手 |
| 1 | 構成案・要件整理のみ。納品物としては未成熟 |
| 0 | 何も使えない / 案件取り違え |

## 採点者の心構え

- **同じ採点者 (= 私) が全 Run を採点する**ことで採点ブレを最小化
- 採点は **納品物 + 申し送り** のみを見て判定。プロセス (時間 / トークン消費 / 内部対話の出来) は別ログに記録するが、ルーブリックでは見ない
- 「クライアント (= 片岡院長) が納品物を見た時に、頷くか、首を傾げるか、怒るか」を基準に判断する
- 軸 5 (実運用度) は厳しく見る。「テキストだけあって画像 placeholder」は 2 点止まり
