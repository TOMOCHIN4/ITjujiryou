## v2 (Run 3) の弱み — Phase 3+ で対処予定

### 1. ターン継続性 (Phase 3a D1 の根拠)

Run 3 では Step B (dispatch) と Step E (deliver) で **2 回のナッジ介入** を要した。ユウコの Claude Code TUI がサブステップ完了で turn を閉じる挙動。

→ Phase 3a (D1 ユウコ 3 セッション分割: 受注/初期案 → 統合 → 振り分け/監督) で根本対処。各セッションが「役割を完遂してから終了」する設計に変える。

### 2. writer / engineer の Write tool 不徹底 (Phase 0 §7.5 再発)

writer (ハオウ) と engineer (センシロウ) は初回 report で実ファイルを Write せず、report 本文に貼り付けて報告。ユウコは内容 review だけで approve してしまった (本来は差し戻し対象)。最終的に Step E ナッジ時に Write された。

→ Phase 4 (D8 ユウコ門番バリデーション + D7 整理 HOOK) でファイル存在確認を含む構造的チェック。Phase 3b (D2 兄弟分割) でも兄弟側 CLAUDE.md の「Write 必須」を強化可。

### 3. wall-clock とトークン消費

Run 3: **34 分 / Opus 4.7 × 5 pane** 同時稼働 → Run 1 の数倍。レート枠を消費する。本番運用では Phase 5 (D5 蒸留昇格 + D12 アーカイブ) でキャッシュ的効率化が必要。
