## 1. 議論の起点 (ユーザー発言)

> 「愛帝十字陵システムの存続に疑問を覚えるようになってきた。」

提起された 4 点:

1. **複雑性の懸念**: 実運用にたどり着けないほどシステムが複雑化。チューニングもきりがなく、今後のメンテナンス不安
2. **資産として残したいもの**: エージェントの記憶・経験を残し育てる仕組み。特化エージェントの専門性
3. **不要かもしれないもの**: オーケストレーターによる自動連携
4. **必要なもの**: 人間 (発注者) と多重人格エージェントの間にワンクッション

## 2. 議論の経過

### 2.1 残す・捨ての仮分類

**残す方向 (仮)**:
- `data/memory/` の積み上げ (会社記憶 + 個人記憶 + 過去案件履歴)
- 各 role の人格・専門知識ファイル群 (`workspaces/{role}/CLAUDE.md` 等)

**捨てる方向 (仮)**:
- tmux 5 pane 並列起動 (`start_office.sh` 等)
- `scripts/inbox_watcher.py` (SQLite poll → tmux send-keys)
- `src/mcp_server.py` の自動連携系ツール (`dispatch_task` / `consult_souther` / `consult_peer`)
- `scripts/hooks/inject_souther_mode.py` 等の自動召喚
- SQLite messages テーブル経由のエージェント間通信
- サザン二重構造 / `[BACKSTAGE:curator]` sentinel 等

### 2.2 モデル固定方針 (本セッション中に確定)

全 5 pane + subagent で **`claude-opus-4-7` + ≥ medium thinking** 固定。Sonnet / Haiku 禁止。

- 関連 memory: `feedback_model_opus_only.md`
- 関連変更: `scripts/use_haiku.sh` / `scripts/use_opus.sh` を削除、`SPEC.md` §4-5「モデル固定」を明文化

### 2.3 ワンクッションの役割 (仮整理、未決)

候補機能:

1. 要件整理 (曖昧な依頼を構造化)
2. ルーティング助言
3. 品質一次レビュー
4. 記憶アクセスの仲介

連携モードの選択肢を 3 つ提示したが、本議論時点では未決:

- (α) ルーター型 — 提案するだけ、人間が手動で専門エージェントを呼ぶ
- (β) ファシリテーター型 — 要件整理と一次レビューを担当、連携はしない
- (γ) トップエージェント型 — 1 プロセス内で subagent / Skill を呼ぶ

### 2.4 「半自律」という訂正 (ユーザー指摘)

私 (Claude) が当初「反自律」として整理していたが、ユーザーから訂正:

> 「私が掲げたいのは反自律じゃなくて半自律だった」

意味:
- **段 (普遍フローの各ステップ内部) は自律で進める**
- **踊り場 (段間の判断点) で人間が介入する**

→ 段の入口 (handoff) と出口 (deliverable) が観察可能なら、subagent の不透明性は許容範囲に入る。連鎖呼び出し (subagent が subagent を呼ぶ) は禁止のまま。

### 2.5 普遍フロー + 踊り場 (ユーザー発案)

異なるドメイン (建築 / レストラン / 医療 / 教育) を横断する「仕事完了までの共通骨格」が存在する、という観察:

```
A. 受信 → B. 構想  → 踊り場 1 (合意確認)
       → C. 設計  → 踊り場 2 (着手前確認)
       → D. 制作  → 踊り場 3 (内部検収)
       → E. 納品  → 踊り場 4 (振り返り)
       → F. 記憶化
```

- 上位 6 段階はドメイン非依存
- 専門性はドメイン文書側で肉付け
- 個別案件は `workflows/cases/{case_id}/` に進捗記録

仮のディレクトリ構造案 (未決):

```
workflows/
  spine/universal_flow.md       # 骨格 (踊り場の定義、不変)
  domains/{domain}.md           # ドメイン別の中身 (建築 / コピー / コード / デザイン / ...)
  cases/{case_id}/              # 個別案件の進捗 (踊り場ごとの判断記録)
```

### 2.6 LLM の盲点 — サプライズ

ユーザー指摘:

> 「LLM は踊り場でのトラブル発覚や人間の心変わりや突然の思い付きといったサプライズ要素を一切想定しない。」

→ Claude (本セッションの私も含む) の根本的弱点。訓練データが完成形に偏り、現場の混乱が書かれない結果。

サプライズへの対処として 2 つの極を私が提示:

- (α) サプライズの余地をなくす = 仕様硬直化、ルーチン案件向け
- (β) サプライズを織り込んだ網の目構想 = 創造案件向け、ただし LLM は経験ベース予測が弱い

→ 私は最初「サプライズスロットを構造化して入れる」「過去事例を列挙する」と提案したが、ユーザーから「多すぎる・細かすぎる・臨機応変すぎる・分からん」と却下。

私の「次の 1 段だけ計画」への揺り戻しも、ユーザーから却下:

> 「それならそれでゴールにもたどりつかないきがするんだよ。つねにゴールも動いちゃうだろ?」

### 2.7 シンプルゴール + N ステップモデル (ユーザー発案 — 議論の山場)

ユーザー発案:

> 「ゴールをシンプル化して、何ステップ以内にたどりつく、というのを初期構想でつくるべきじゃないだろうか。ステップの踊り場ごとにゴールまでの道筋とこれまでの道のりを振り返り、残り工程数でたどり着ける最良のゴールの詳細をイメージしなおし目指し続ける。シンプルゴールはかえない。」

**整理**:

- **固定するもの**: シンプルゴール (案件の達成目標を 1 文) + ステップ数 N (初期構想で確定、変更不可)
- **動かすもの**: 各踊り場で残工程数を見て再イメージする「ゴールの詳細像」

→ サプライズは「詳細像を更新する材料」として受け止め、ゴール本質と N は不変。これにより:
- ドリフトしない (N が予算制約として効く)
- 方向喪失しない (シンプルゴールが anchor)
- サプライズを吸収できる (詳細像が可変)

ユーザー追加コメント:

> 「これこそがオーケストレーターの仕事だろう。」

→ ワンクッション (オーケストレーター) の核心業務は、踊り場ごとに「シンプルゴールに対して残 (N-X) 段で到達できる最良の詳細像」を再構築することだと位置付けられた。

### 2.8 スコープの最終確認 (ユーザーから明示)

> 「シンプルゴールやゴール詳細というのは受注案件ごとの達成目標のことです」

→ シンプルゴール / N / 詳細像モデルは **個別案件単位の概念**。システム改修プロセス自体への適用は混同 (私の側の拡張)、議論本筋から除外。

## 3. 仮の合意 (議論段階、確定ではない)

以下は議論の流れから抽出した「方向性として共有された」内容で、まだ実装計画化されていない:

- システムは自前ハーネスを撤去し、Claude Code 標準機能 + markdown + filesystem memory に縮減
- 半自律モデル (段は自律、踊り場は人間)
- 普遍フロー骨格 (A→B→C→D→E→F + 踊り場 1-4) を導入、ドメイン文書で肉付け
- 案件単位で「シンプルゴール + N + 詳細像」を管理、踊り場ごとに詳細像を更新
- subagent は段 D に限定使用 (連鎖禁止、段の入口と出口を観察可能に保つ)
- 計画は薄く、システムは捨てやすく、起きたことを memory に残す
- モデルは Opus 4.7 + ≥ medium thinking 固定 (本セッション中に確定済)

## 4. 未決の論点

### 4.1 設計レベル

- **N の決め方**: 案件規模で可変 (小 3 / 中 5 / 大 7) か固定 (常に 5) か
- **N に収まらない時の挙動**: 詳細像を畳む → それでも無理なら案件中止、で原則 OK か
- **シンプルゴール自体が誤りと判明した時**: 案件仕切り直し / 中止のどちらか
- **踊り場の判断パターン**: 強型 (人間判断のみ) vs 弱型 (ワンクッション提案 → 人間承認) を踊り場ごとに振り分けるか統一するか
- **専門エージェントの呼ばれ方**: (a) 指示書 + 人間が `/role` 実行、(b) Task tool で subagent 同期呼び出し、(c) 別 Claude Code セッション手動起動

- **撤退対象の確定**: memory 層 + persona ファイル群以外はほぼ全て捨てる方向だが、`feat/yuko-3pipe` ブランチをどうするか
