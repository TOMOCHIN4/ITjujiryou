# Phase A プラン

> **作成日**: 2026-05-17
> **シンプルゴール (フロー全体)**: `prompts/` 配下の全ファイルの索引 + 1 行要約を `docs/prompts_overview.md` として整備し commit 済みにする
> **全 Phase 数 N**: 3
> **当 Phase の位置**: A / 3

## 1. 背景

`prompts/` 配下には 7 ファイル (`_company_motto.md` / `designer.md` / `engineer.md` / `souther_president.md` / `souther_quotes.md` / `writer.md` / `yuko_secretary.md`) のプロンプトテンプレートが存在するが、各ファイルがどのキャラ/役割向けで何の責務を持つのか索引が無く、新規参加者や本セッション側からも一覧で見渡せない。

`docs/prompts_overview.md` を新設し、ファイル名 / 役割 / 1 行要約をテーブル化して索引化する。フロー全体は 3 Phase 構成:

- **Phase A (本 Phase)**: `prompts/` 全ファイル読了 + 1 行要約・用途タグの草案作成
- **Phase B**: 草案を `docs/prompts_overview.md` として書き出し (Markdown 構造化)
- **Phase C**: 内容検証 + commit

Phase A は「調査 + 要約方針合意」までで、ファイル書き出しは行わない。

## 2. 当 Phase の完了判定

- [ ] `prompts/` 配下 7 ファイルの内容を全て読了
- [ ] 各ファイルについて以下を草案として作成
  - ファイル名
  - 役割 / 対象 (キャラ名 or 横断用途)
  - 1 行要約 (80 文字目安)
  - 用途タグ (例: `system-prompt` / `quote-library` / `reference`)
- [ ] 草案をユーザーに提示し、要約粒度 / タグ体系 / 並び順について合意

## 3. 成果物

- `prompts/` 全 7 ファイルの 1 行要約 + 用途タグの草案 (会話内テキスト、ファイル書き出しは Phase B)
- ユーザー合意済みの「要約方針」(粒度・タグ体系・並び順)

## 4. 想定リスク

| リスク | 対処 |
|---|---|
| 1 行要約が長くなる / 抽象的になる | 80 文字目安を堅持、収まらないファイルは独立に確認 |
| 用途タグの分類軸が曖昧 | 草案時に最大 4 タグまでに収め、ユーザー確認で確定 |
| `_company_motto.md` のような特殊ファイルの扱い | アンダースコア prefix = 横断引用素材として別カテゴリ化を提案 |
| 機微情報 (社内のみ有効な世界観) の docs/ 露出 | docs/ は社内向け開発ドキュメント前提で進める、外部公開資料化はしない |
