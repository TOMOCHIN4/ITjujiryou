
# 愛帝十字陵 v2 議論結果

最終更新: 2026-05-12

本書は v2 の **設計思想・運用ルール** のみを記述する。実装機構 (tmux 構成、プロセス間通信、ディレクトリ物理配置、起動スクリプト等) は既存実装に従い、本書では扱わない。

---

## 0. システムの目的と優位性

- **目的**: 最終制作物の高品質化。`Opus 4.7 + 厳選SKILLS` 単体を上回る納品物を出すこと。
- **中核の優位性**: コンテキストの分離。5つの独立Claude Codeプロセスにより、単体運用では避けられない混線を構造で抑え込む。
- **副次的な価値**: ペルソナによる役割の固定。各キャラの気質が品質ガードレールとして機能する。

---

## 1. 記憶システム

### 1.1 記憶の種類

- **会社記憶**: 全員共通。会社としての経験を蓄積する。
- **個人記憶**: 本人のみ書き込み可。個人としての経験を蓄積する。

### 1.2 アクセス権

| キャラ | 会社記憶 | 自身の個人記憶 | 他人の個人記憶 |
|---|---|---|---|
| ユウコ | 読み | 読み書き | **全閲覧可** |
| サザン | 読み書き (HOOK経由) | 読み書き | 不可 |
| 三兄弟 | 読み | 読み書き | 不可 |

ユウコだけが全記憶を読める特権を持つ。その代償としてコンテキスト膨張のリスクを背負う。

### 1.3 案件中の積み重ね方

- 案件中は各人がただ積み上げる。
- 案件終了時に整理HOOKが各兄弟で発火し、`self/memory` への追記と `company/memory` への提案を生成する。

---

## 2. サザンの設計

### 2.1 ペルソナ原則

- 制圧前進あるのみ。ユウコの計画に**絶対反対しない**。
- 具体的な案は求めない。**相談ではない**。
- 三兄弟との**コミュニケーションは一切ない**。ユウコとだけ対話する。

### 2.2 技術的役割

- 発言は儀礼承認のみ (「制圧前進」「うむ」「よかろう」等の固定文言)。
- セッション完了が `company_memory_write` HOOK のトリガとなる。
- **HOOK が物理ファイル更新を担当**。サザンは声、HOOKは権能。

### 2.3 嘘をつかせない設計

- 兄弟の提案は **ユウコ側でバリデーション・矛盾解決まで完了させる**。
- クリーンな提案だけがサザンに届くため、HOOKは黙々と反映するだけ。
- サザンが「制圧前進」と発した時、それが実装上の嘘になることはない。

## 3. ユウコのセッション設計

### 3.1 3セッション分割

| セッション | 役割 | 引き継ぎ先 |
|---|---|---|
| 受注 / 初期案 | grill-me 等で要件確定、全記憶 + WF庫を参照しイニシャルプラン作成 | → 統合へ「要件 + 初期案」 |
| 統合 | 初期案 + 3応答を統合、ファイナルプラン確定、サザン儀礼承認を受ける | → 振り分けへ「ファイナルプラン (確定MD)」 |
| 振り分け / 監督 | マクロフロー + 各人のミクロワークフローを層構造で保持、兄弟プロセスへ仕事を投下、成果物を集約 | — |

### 3.2 ノード視点

ユウコにとっての **ノードは各社員**。誰にいつ仕事を渡し、誰から何を受け取り、どこへ流すか。直列 / 並列 / ループの合成を決める。

---

## 4. 三兄弟のセッション設計

### 4.1 セッション分離

- 各兄弟は独立プロセスとして走る (短命セッション)。
- レビュー段階と実行段階で **別セッション** として立てる。

### 4.2 段階別の入力と振る舞い

| 段階 | 入力 | 出力 | 個性 |
|---|---|---|---|
| レビュー | マクロフロー全体図、自分のミクロWF候補、自分の company/memory と self/memory | 承認 or 修正案 + **理由を self/memory に書き込み** | 反映される |
| 実行 | ファイナルプランから自分の担当ノードのみ、案件中蓄積された自分の self/memory | 成果物 | 出さず計画通りに動く |

### 4.3 ハンドオフの方針

- 同一兄弟のレビュー → 実行ハンドオフは **self/memory 経由**。
- 案件中の self/memory 肥大化は当面許容。許容できないレベルになれば再検討。

### 4.4 ノード視点

三兄弟にとっての **ノードは skill / MCP**。渡されたサブタスクを、事前用意したワークフローから選び微調整して実行する。

## 5. 業務フロー全体

```
顧客発注
  ↓
ユウコ: 受注・要件定義 (grill-me 等)
  ↓
ユウコ: イニシャルプラン作成 (全記憶 + WF庫参照)
  ↓
三兄弟 並列レビュー (承認 / 修正案、理由を self/memory に書込)
  ↓
ユウコ: ファイナルプラン確定 (統合セッション)
  ↓
サザン: 儀礼承認「制圧前進」
  ↓
ユウコ: 振り分け (直列 / 並列 / ループ混合)
  ↓
三兄弟: ワークフロー実行
  ↓
納品 → 顧客
  ↓
各兄弟: 整理HOOK (self/memory 追記 + company 提案)
  ↓
ユウコ: 統合・バリデーション (門番)
  ↓
サザン: 儀礼承認 → company_memory_write HOOK → 物理反映
```

## 6. 指示の粒度

- **ワークフロー名指定 + どこをどう改変するかの本文指示**。
- skill 名指定までしない。
- 初期案・レビュー・実行すべて同じ語彙 (ワークフロー) で噛み合うようにする。

### 例

> ユウコ → トシ: `build-feature-v3` をベースに、ステップ4の `integration-test` を本案件のAPI仕様に合わせて差し替え。ステップ7はスキップ可。

> トシ → ユウコ: `build-feature-v3` はOK。ただしステップ4より `build-feature-v4` の方が本件適合。理由は `self/memory/2026-03-xx` の知見による。

## 7. ワークフローと層構造

### 7.1 形式

- ワークフローもファイナルプランも、Claude Code が読める Markdown + YAML frontmatter 形式。
- **カスタムスキーマは発明しない**。Studio が吐く形式 = Claude Code が読む形式に統一する。
- ファイナルプランの本文は自然言語のマクロフロー記述。実行エンジンは作らず、ユウコのClaude Codeセッションが本文を解釈して兄弟へ仕事を投下する。

### 7.2 層構造

ユウコは案件中、層構造で保持する:

- **マクロ層**: 誰にいつ仕事を渡し、誰から何を受け取り、どこへ流すか。直列 / 並列 / ループの合成。
- **ミクロ層**: 各兄弟が使うワークフローの中身、改変指示。

ファイナルプランは両層を含む1枚の MD として書かれる。

---

## 8. ワークフロー倉庫の3階層

| 階層 | 用途 | 改変可否 |
|---|---|---|
| 原本 (originals) | Studio 出力、人間が手で配置 | **改変禁止** |
| 案件アレンジ (cases) | 案件ごとのアレンジ、案件のスナップショット | 案件中のみ書き換え |
| 蒸留 (distilled) | 案件後の昇格版、再利用可 | 昇格時のみ書き込み |

### 8.1 昇格規律

- 案件終了時の整理HOOKで各兄弟が「再利用価値あり」と判定した1本のみ昇格提案。
- ユウコが承認したものが蒸留層に昇格。
- **蒸留昇格は1案件・1兄弟あたり1本まで**。
- 蒸留層は同じ原本から派生3本上限。4本目を作る時は1本退役。

### 8.2 肥大化制御

- 原本は触らない。Studio のバージョンアップは `v3` → `v4` として共存。
- 案件アレンジは90日でアーカイブ (物理削除はせず退役領域へ移動)。
- 原本も90日無参照で削除候補に。

## 9. Workflow Studio との連携

### 9.1 ツールの位置づけ

- 外部ツール (VSCode 拡張 CC Workflow Studio 等)。
- ワークフローを `.md` のスラッシュコマンドまたは agent skill としてエクスポート可能。

### 9.2 運用ルール

- **業務時間外に人間が操作**。AIエージェントには操作させない。
- 案件が走っている間は触らない。
- 出力先 (原本層) への配置は人間が手動で行う。
- ユウコが生成したファイナルプランを Studio で開いて編集することも可能 (同じ形式のため)。

## 10. company/memory 書き込みフロー

```
各兄弟の整理HOOK
  ↓ (提案ファイル生成)
ユウコ統合セッション (門番役)
  ↓ フォーマット検証 + 矛盾解決 + 粒度調整 + 差戻し
  ↓ クリーン提案だけが下流へ
サザンセッション (儀礼承認のみ発言)
  ↓ セッション完了がトリガ
company_memory_write HOOK
  ↓ 物理反映
company/memory 更新完了
```

- バリデーションはユウコ側で完結する。
- サザンに届く時点で **必ず可決される状態**。
- 兄弟とサザンの間に直接の経路は存在しない。

---

## 11. 設計上の重要原則

### 11.1 パーサ・実行エンジンは作らない

ユウコの振り分けセッション (Claude Code セッション) がファイナルプランの本文を読み、自然言語の指示通りに兄弟プロセスへ仕事を投下する。LLMがマクロ記述を解釈する。状態管理は既存実装に従う。

### 11.2 すべてのファイルは debuggable

全部 MD で人間が読める。「なぜこう動いた」の追跡が grep だけで済む。プラン肥大化も目視で分かる。

### 11.3 Claude Code 進化に自動追従

スラッシュコマンドや subagent の仕様が拡張されれば、IT十字陵もそのまま恩恵を受ける。Studio が新機能を吐けば、それも自動的に使える。

### 11.4 ユウコ起点のバリデーション

`company/memory` への書き込みは、ユウコの統合セッションが門番として機能する。差戻しはユウコで完結し、サザンには必ず可決される状態の提案だけが届く。

## 13. 確定回答 (Q1〜Q3 + 補足)

実装側評価書の3問への最終回答 (2026-05-12)。

### 13.1 Q1: セッション分離の物理形 → (α) ステートレス短命

`claude -p <prompt>` を都度起動する完全ステートレス方式を採用。pane を持たず、各セッションはバックグラウンドプロセスとして走る。フェーズ間ハンドオフは既存の inbox/outbox ファイル基盤を流用 (ユウコの 3 セッションが同 pane で順次起動する場合、前セッションの成果物 — initial plan / integrated plan / final plan — を所定パスに書き、次セッションが起動時に読む)。

(B) モード切替案は採用しない (v2 の核「コンテキスト分離による品質向上」が消えるため)。(A) pane 増設案も採用しない (Max OAuth と画面で破綻)。

### 13.2 Q2: メモリ物理配置 → 6 ディレクトリ構成

```
data/memory/company/         # 全員読み、サザン HOOK のみ書き
data/memory/souther/         # サザン個人記憶 (読み書き本人のみ、ユウコは閲覧可)
data/memory/yuko/            # ユウコ個人記憶
data/memory/writer/          # ハオウ個人記憶
data/memory/designer/        # トシ個人記憶
data/memory/engineer/        # センシロウ個人記憶
```

設計書 §1 (人物表記: サザン / ユウコ / 三兄弟) および Q2 回答例 (sazan/haou/toshi/senshirou) との対応:

| 設計書表記 | 実装パス |
|---|---|
| サザン | `souther/` |
| ユウコ | `yuko/` |
| ハオウ | `writer/` |
| トシ | `designer/` |
| センシロウ | `engineer/` |

実装パスはコード識別子 (MCP server / settings.json / inbox-watcher で使用中) に統一。Claude Code 組み込み auto-memory (`~/.claude/projects/.../memory/`) は人間 (= ユーザー主セッション) の私的領域として継続、本会社エージェント網とは別レイヤー。

### 13.3 Q3: Workflow Studio → 不採用 (スキーマ互換のみ確保)

CC Workflow Studio (breaking-brake/cc-wf-studio) は実在の VSCode 拡張だが、現時点で導入しない。原本ワークフロー MD は人間が手書きする。

ただし将来 Studio を採用する時に移行コストがゼロになるよう、以下を揃える:
- ワークフロー MD の frontmatter フィールドを Studio エクスポート形式に合わせる
- `workflows/originals/` の配置を Studio の出力先パス規約に合わせる
- ファイナルプランも同形式 (slash command スタイル) で書く

frontmatter スキーマの具体的なフィールド確定は次セッションの宿題 (§19)。

## 14. サザンの二重構造 (D9 再定義)

D9 を **実装簡素化** ではなく **役割の再定義** として扱う。

### 14.1 表向きの役割

ユウコからの伝令に対し、サウザー名台詞 21 選から 1 つを選んで本案件文脈にオマージュさせた応答を返す。**それだけ**。

- 具体的な検討・修正・提案は一切しない
- 三兄弟との直接対話は依然としてゼロ (ユウコ経由のみ)
- プロンプト指示は「名台詞 21 選 + オマージュ + 案件文脈」レベルで足りる

`v0.13 brevity hook` (英語化禁止 + sentence cap 注入) は **Phase 1 で退役**。サザンの発話形式が「21 選オマージュ」固定になるため、追加ガードレールは不要になる。

### 14.2 裏側の役割

サザンを儀礼承認だけのために独立エージェントとして存続させるのはリソースの無駄遣い、かつユウコに兼任させるとコンテキストが膨張する。両方を避けるため、**「全体タスクのサブエージェント的存在」** として裏で動かす。

裏側に割り当てる内部タスクの条件: **案件中に必ず発生し、文脈をほぼ持たなくても処理できる単純な全体タスク**。

確定済み HOOK は `company_memory_write` の 1 点のみ。他は **Phase 0 Track C** で過去ログから逆引きして決定する。実装側で列挙した候補 (Phase 0 着手後に精査):

| ユウコ → サザン報告タイミング | 物理イベント | サザン裏 HOOK 候補 |
|---|---|---|
| 新規案件受注承認上申 | tasks 行追加、client_request 確定 | client 入力サニタイズ + 過去類似案件検索 (将来) |
| 値引き / 納期短縮の裁定 | クライアント再交渉メッセージ受信 | (純儀礼で良いかも) |
| 品質基準で迷う成果物の上申 | revisions に escalate_to_president 行追加 | 直近 N 件の判断履歴を company/memory に集約 |
| ファイナルプラン承認 (v2 新規) | プラン MD 確定 | プラン MD の構造バリデーション (frontmatter チェック) |
| company 提案承認 (v2 新規) | クリーン提案がキュー入り | **`company_memory_write` 反映 (確定)** |
| 案件完了報告 (v2 新規) | deliver 完了 | 整理 HOOK 起動 (`post_deliver_hook.py` で各兄弟の self/memory 整理を促す) |

### 14.3 同期 polling との関係

現状 `consult_souther` は 60 秒 polling で同期。新サザンは応答が「21 選オマージュ」と短いため数秒で返り、polling 自体は維持で問題なし。**ただし「応答受領 ≠ HOOK 完了」** を明文化する: ユウコは応答を受け取った時点で次セッションへ進んでよく、HOOK の物理処理完了は気にしない。ユウコの統合セッションが終わって振り分けセッションが始まる頃には HOOK 処理は完了している、という時間軸前提。

## 15. Phase 0 セットアップ詳細

### 15.1 ベンチマーク案件: かたおか歯科クリニック歯科衛生士求人セット

1 件のロングラン複合案件として固定。

| 項目 | 内容 |
|---|---|
| 求人者 | 大阪府枚方市 かたおか歯科クリニック |
| 求人対象 | 歯科衛生士 |
| ターゲット読者 | 徳島大学歯学部 口腔保健科の学生 (卒後は歯科衛生士) |
| 想定納品物 | LP、パンフレット、ポスター 他 (確定範囲は案件中 grill-me で決定) |

### 15.2 Phase 0 で確定する事項

- **顧客役は人間 (ユーザー本人)** が毎 Run で担う
- **初期発注書 fixture**: `benchmarks/cases/kataoka-dental/initial_request.md` に 300-500 字程度の発注メッセージを人間が用意、毎 Run の起点
- **顧客ペルソナ**: `benchmarks/cases/kataoka-dental/client_persona.md` に顧客役の一貫したキャラメモを人間が用意

### 15.3 Phase 0 では確定しない事項 (案件中に grill-me で決まる)

- 納品物の最終リスト
- デザイン物の出力形式
- ブランドカラー / トーン&マナー / 訴求軸

grill-me 能力自体が評価軸の一部になる。

### 15.4 検収サイクル
